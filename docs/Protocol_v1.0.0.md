# Спецификация протокола JuThesis v1.0.0

## Введение
Данный документ описывает протокол обмена данными JuThesis версии 1.0.0. 
Протокол определяет формат входных и выходных файлов для системы оптимизации тестового покрытия и обеспечивает независимость от конкретного языка программирования или системы тестирования.
Протокол спроектирован с учетом следующих принципов:
1. Протокол должен быть языконезависимым и поддерживать любые системы тестирования через единый формат данных. 
2. Протокол должен быть расширяемым с поддержкой версионирования для обеспечения обратной совместимости.
3. Протокол должен содержать только необходимую информацию для работы алгоритма оптимизации. 
4. Протокол должен быть человекочитаемым для упрощения отладки и создания входных файлов вручную.

## Версионирование
Протокол JuThesis использует семантическое версионирование в формате MAJOR.MINOR.PATCH.
Текущая версия протокола - 1.0.0.
Версия протокола указывается в поле `version` как во входных, так и в выходных файлах.
При изменении формата данных версия протокола обновляется следующим образом.
MAJOR версия увеличивается при внесении несовместимых изменений, требующих обновления всех клиентов и плагинов.
MINOR версия увеличивается при добавлении новой функциональности с сохранением обратной совместимости.
PATCH версия увеличивается при исправлении ошибок без изменения формата данных.

Все реализации JuThesis должны проверять версию протокола при чтении входных файлов и отклонять файлы с неподдерживаемой версией.

## Формат входных данных

Входной файл представляет собой JSON документ, содержащий информацию об измененных функциях, доступных тестах и параметрах оптимизации.

### Структура входного файла
```json
{
  "version": "1.0.0",
  "modified_functions": [...],
  "available_tests": {...},
  "time_budget": 0.0,
  "max_initial_coverage_size": 0
}
```

### Поле version
Обязательное строковое поле, указывающее версию протокола.
Для данной спецификации значение должно быть равно `"1.0.0"`.
Реализации должны проверять это поле при чтении файла и отклонять файлы с неподдерживаемыми версиями.

### Поле modified_functions
Обязательное поле, представляющее собой массив строк с уникальными идентификаторами изменённых функций или методов.
Каждый элемент массива - это произвольная строка, однозначно идентифицирующая функцию в контексте проекта.

Формат идентификатора определяется пользователем или плагином JuThesis.
Протокол не накладывает ограничений на формат идентификатора, главное требование - уникальность в пределах одного входного файла.

Пример:
```json
{
  "modified_functions": [
    "func1", "func2", "func3", 
    "func4", "func5", "func6"
  ]
}
```

Массив не должен содержать дубликатов.
Пустой массив технически допустим, но приведет к тривиальному решению без выбора тестов.

### Поле available_tests
Обязательное поле, представляющее собой объект (словарь), где ключи - это уникальные идентификаторы тестов, а значения - объекты с информацией о каждом тесте.
Формат идентификатора теста аналогично идентификатору функции определяется пользователем или плагином.

Каждый объект теста содержит два обязательных поля:
- `time` - число с плавающей точкой, представляющее среднее время выполнения теста в секундах. Значение должно быть строго положительным (больше нуля).
- `covered_functions` - массив строк, содержащий идентификаторы функций, которые покрывает данный тест. Каждый идентификатор должен ссылаться на функцию из массива `modified_functions`, хотя протокол допускает и другие функции (они будут проигнорированы при оптимизации).

Пример:
```json
{
  "available_tests": {
    "test1": {
      "time": 3,
      "covered_functions": ["func1", "func2", "funcA"]
    },
    "test2": {
      "time": 4,
      "covered_functions": ["func2", "func3", "func4", "funcB"]
    },
    "test3": {
      "time": 2,
      "covered_functions": ["func4", "func5", "funcC" ]
    },
    "test4": {
      "time": 3,
      "covered_functions": ["func3", "func6", "funcD"]
    }
  }
}
```

Ключи объекта `available_tests` не должны дублироваться.
Пустой объект допустим технически, но приведет к решению без выбора тестов.

### Поле time_budget
Обязательное числовое поле, представляющее общий бюджет времени на выполнение тестов в секундах.
Значение должно быть строго положительным (больше нуля).

Бюджет времени определяет максимальное суммарное время выполнения всех выбранных тестов. 
Алгоритм оптимизации гарантирует, что суммарное время выбранных тестов не превысит указанное значение.

Пример:
```json
{
  "time_budget": 7
}
```

### Поле max_initial_coverage_size
Обязательное целочисленное поле, определяющее максимальную мощность начальных покрытий, перебираемых алгоритмом. Обозначается в математической модели как K. 
Значение должно быть неотрицательным целым числом.

Алгоритм JuThesis перебирает все возможные начальные покрытия мощности от 0 до K включительно.
Увеличение K улучшает качество решения, но экспоненциально увеличивает время работы алгоритма.
Рекомендуемое значение для большинства практических задач - 2.

Пример:
```json
{
  "max_initial_coverage_size": 2
}
```

### Полный пример входного файла
```json
{
  "version": "1.0.0",
  "modified_functions": [
    "func1", "func2", "func3", 
    "func4", "func5", "func6"
  ],
  "available_tests": {
    "test1": {
      "time": 3,
      "covered_functions": ["func1", "func2", "funcA"]
    },
    "test2": {
      "time": 4,
      "covered_functions": ["func2", "func3", "func4", "funcB"]
    },
    "test3": {
      "time": 2,
      "covered_functions": ["func4", "func5", "funcC" ]
    },
    "test4": {
      "time": 3,
      "covered_functions": ["func3", "func6", "funcD"]
    }
  },
  "time_budget": 7,
  "max_initial_coverage_size": 2
}
```

## Формат выходных данных
Выходной файл представляет собой JSON документ, содержащий результат работы алгоритма оптимизации - список выбранных тестов и статистику покрытия.

### Структура выходного файла
```json
{
  "version": "1.0.0",
  "tests": [...],
  "total_execution_time": 0.0,
  "total_functions_covered": 0
}
```

### Поле version
Обязательное строковое поле, указывающее версию протокола выходного файла.
Должно совпадать с версией входного файла.
Для данной спецификации значение равно `"1.0.0"`.

### Поле tests
Обязательное поле, представляющее собой массив объектов с информацией о выбранных тестах.
Каждый объект содержит два обязательных поля:
- `test` - строка с идентификатором выбранного теста. Это один из ключей из объекта `available_tests` входного файла.
- `functions` - массив строк с идентификаторами функций, которые фактически покрываются данным тестом. Важно отметить, что это подмножество покрытия теста, включающее только функции из массива `modified_functions` входного файла.

Массив упорядочен в том порядке, в котором тесты были выбраны алгоритмом.

Пример:
```json
{
  "tests": [
    {"test": "test1", "functions": ["func1", "func2"]},
    {"test": "test3", "functions": ["func4", "func5"]}
  ]
}
```

Пустой массив означает, что не удалось выбрать ни одного теста (например, если все тесты превышают бюджет времени).

### Поле total_execution_time
Обязательное числовое поле, содержащее суммарное время выполнения всех выбранных тестов в секундах. 
Значение вычисляется как сумма времен выполнения тестов из массива `tests`.
Это значение меньше или равно `time_budget` из входного файла.
Если выбранных тестов нет, значение равно 0.0.

Пример:

```json
{
  "total_execution_time": 5.0
}
```

### Поле total_functions_covered
Обязательное целочисленное поле, содержащее общее количество покрытых функций.
Это значение целевой функции оптимизации - именно его максимизирует алгоритм JuThesis.
Значение вычисляется как количество уникальных функций, встречающихся в массивах `functions` всех выбранных тестов. 
Значение не превышает количество функций в массиве `modified_functions` входного файла.

Пример:
```json
{
  "total_functions_covered": 4
}
```

### Полный пример выходного файла
```json
{
  "version": "1.0.0",
  "tests": [
    {"test": "test1", "functions": ["func1", "func2"]},
    {"test": "test3", "functions": ["func4", "func5"]}
  ],
  "total_execution_time": 5.0,
  "total_functions_covered": 4
}
```
